# 構造化BASICエディタ ＋ MSX-BASIC変換システム 設計書

本書は、**構造化BASIC**を入力とし、**MSX-BASIC** へ変換するエディタ兼プリプロセッサの
全体設計・内部構造・データモデル・変換アルゴリズム・UI仕様・保存処理フローを定義する。
本READMEは **プロジェクトの全体像を正確に伝える入口** であり、各論は番号付きドキュメントへ委譲する。

---

## 1. プロジェクト概要

| 項目 | 内容 |
|------|------|
| 目的 | 構造化BASIC（FUNCTION・ブロック構造・REF引数・BREAK/CONTINUE）で記述したコードを、MSX-BASIC（行番号・GOSUB・GOTO）へ機械変換する |
| 入力 | 構造化BASIC ソース（`.msxb`） |
| 出力 | MSX-BASIC ソース（`.bas`）＋ 変換テーブル（`.map.json`） |
| 実行環境 | **クロスプラットフォーム・デスクトップアプリ（Tauri / Windows・macOS両対応）**。エディタ・変換・保存は両OS共通 |
| 再生環境 | 再生は3方式：**A.公式MSXプレイヤー連携（Windows専用）／ B.WebMSXリンク＋コピペ（Win/Mac、URL設定可）／ C.外部エミュレータ＋仮想FD(.dsk)（Win/Mac、openMSXはMac対応）**。Macもネイティブ再生はCで可能（[10](10-tech-stack.md)） |
| ターゲット機 | MSX2 以降、R800/R80（turboR）での実行を前提に最適化 |

### 設計思想（3原則）

1. **ネストを自由に許可する** — ブロック（IF/FOR/WHILE）の任意の入れ子を許す（FUNCTIONのネストのみ禁止）。詳細な理由は §2。
2. **可逆変換を原則とする** — 変換テーブルにより MSX-BASIC → 構造化BASIC を復元できる。役割は §5。
3. **作業内容を失わせない** — 文法エラーがあっても変換前ソースは必ず保存可能とする（§6 EditorUI／[08](08-file-save.md)）。

### 構造化BASICの最小例（言語の特徴を一望する）

この短い例に、本言語の主要素 — **FUNCTION / REF引数 / ネストしたIF / FOR…NEXT / どこでもRETURN / コメント保持** — がすべて含まれる。

```basic
' 配列の中から最初に 0 を見つけて返す
FUNCTION FIND_ZERO(REF IDX)
    GLOBAL A                   ' 配列Aはグローバル → 明示宣言（PHP流。§2参照）
    FOR I = 1 TO 10
        IF A(I) = 0 THEN
            IDX = I
            RETURN 1
        END IF
    NEXT I
    RETURN 0
END FUNCTION

DIM A(10)
A(3) = 0
RESULT = FIND_ZERO(WHERE)
PRINT "FOUND="; RESULT; " AT "; WHERE
```

- `REF IDX`：参照渡し。呼び出し側 `FIND_ZERO(WHERE)` では REF を省略でき、参照渡しかどうかは関数シグネチャで判定する（[01](01-language-spec.md#133-引数渡し仕様1-3)）。
- `GLOBAL A`：トップレベルの配列 `A` を関数内で使う宣言。**関数内は既定ローカル、共有は `GLOBAL` で明示**（QuickBASIC/PHP流、[01 §1.10](01-language-spec.md#110-変数スコープ)）。`I` は宣言不要＝ローカル。
- `IF` が `FOR` の中にある＝**ネスト**（許可。§2）。`RETURN 1` は関数の途中＝**どこでもRETURN**。
- 代入の `LET` は省略可、`DIM`・配列添字も使用できる（[01](01-language-spec.md#19-代入let省略配列dim)）。
- 文字列リテラルの引用符は **ASCII の `"`** のみ（スマートクォート禁止）。`PRINT` の連結区切りは **`;`（セミコロン）**（[§9](#9-変換後コードの品質要件)）。

## 2. 設計判断：なぜネストを「許可」するのか

> **【重要な方針転換の記録】** 当初は原仕様1-4に従い「ネスト禁止」で設計していたが、
> 変換後コードのサイズ・メモリ・実装容易性を評価した結果、**完全ネスト許可**へ方針転換した。
> 本書全体はネスト許可を前提に記述している。撤回した「ガード節」「後置IF脱出文」案は採用しない。

ネスト許可は単なる「行数削減」ではなく、以下の **複数の技術的根拠** に基づく決定である。

| 観点 | ネスト許可が優れる理由 |
|------|------------------------|
| **ASTの自然さ** | ブロック本体を `statement` の再帰で表現でき、AST が構造をそのまま反映する。ネスト禁止だと「ブロック内ブロックを弾く」例外規則が必要で、文法・パーサーが歪む |
| **変換の単純化** | MSX-BASIC は `FOR…NEXT` と1行 `IF` を**ネイティブに入れ子可能**。ネストをそのまま入れ子の FOR/NEXT・IF へ落とせ、GOTO/GOSUB の展開が単純になる |
| **行数・メモリ削減** | ネスト禁止では入れ子ロジックを FUNCTION へ切り出すしかなく、呼び出しごとに「引数コピー・GOSUB・戻り値受け・REF書き戻し・専用変数」のボイラープレートが増える。これが行数と変数テーブルを膨張させる（例：配列最大値 ＝ 禁止6行+変数2個 vs 許可3行）。詳細比較は [09](09-optimization.md#92-ネスト許可への方針転換と早期リターン仕様8-2の再解釈) |
| **逆変換の安定性** | ネイティブ FOR/NEXT・IF範囲をスタックで対応付けて再帰復元できるため、逆変換が決定的かつ安定する（[06](06-reverse-transformer.md)） |
| **早期リターンとの両立** | ネストした IF 内からの `RETURN`/`BREAK`/`CONTINUE` を許すことで、早期リターン主体のスタイルと矛盾なく共存する |

> **トレードオフ**：ネスト許可で増えるのは**変換器・パーサーの実装複雑さ**（ブロックスタック、BREAK/CONTINUE のループスタック解決）であり、**出力サイズはむしろ減る**。

### 変換後 MSX-BASIC の最小例（§1の例を変換した出力）

§1の `FIND_ZERO` を変換するとこうなる。行番号昇順・重複なし・CRLF、文の区切りは **`:`（コロン）**。

§1の `FIND_ZERO` を変換した出力。**MSX変数名は先頭2文字のみ有効**なので、グローバル `RESULT`→`RE`・`WHERE`→`WH`、
ローカルは `I`→`FI`・戻り値→`FR` のように **2文字名アロケータ**が割り当てる（[§5](#5-変換テーブルの役割可逆変換の心臓部)・[05 §5.11](05-transformer.md#511-2文字msx名アロケータ全変数全型)）。

```basic
100 ' === MAIN ===
110 DIM A(10)
120 A(3)=0
130 GOSUB 1000: RE=FR
140 PRINT "FOUND=";RE;" AT ";WH
150 END
1000 ' === FUNCTION FIND_ZERO (IDX->PO) ===
1010 FOR FI=1 TO 10
1020 IF A(FI)=0 THEN PO=FI: FR=1: RETURN
1030 NEXT FI
1040 FR=0
1050 RETURN
```

このわずかな出力に、変換器の要点が表れている。**そのまま WebMSX に貼り付けて実行でき**、`FOUND=1 AT 3` を出力する。

- **2文字MSX名** — `RESULT`→`RE`、`WHERE`→`WH`、ローカル `I`→`FI`、戻り値→`FR`。配列 `A` は1文字でそのまま。
- **行番号セグメント** — MAIN は `100,110,…`、関数は `1000,1010,…` の別セグメント（§10.1）。
- **文の区切りは `:`（コロン）** — `PO=FI: FR=1: RETURN`。`;` は `PRINT` の連結専用、`,` はタブ区切りで、文の連結には使えない。
- **REF引数は名前置換（ゼロコピー）** — `FIND_ZERO(WHERE)` の `REF IDX` を**呼び出し側の実変数 `WHERE`（=`WH`）へ直接置換**。
  1020行で `PO` を直接書き換える（コピーは一切無い）。スカラ・配列とも同一規則（[05 §5.4](05-transformer.md#54-ref引数--名前置換方式仕様3-3)）。
- **どこでもRETURN → `戻り値変数=値 : RETURN`** — `FR=1: RETURN` / `FR=0: RETURN`。
- **ネストしたIFが1行IFに落ちる** — 1020行：本体が早期RETURNで終わるため `IF…THEN …:…:RETURN` に畳み込む（§10.2／[05](05-transformer.md#551-ifブロック)）。

> 変換テーブル（`.map.json`）には `IDX↔WHERE`(REF置換)・`RESULT↔RE`/`WHERE↔WH`/戻り値`↔FR`(2文字名対応)・行番号↔ラベルが
> 記録され、これらを逆変換で復元できる（§5）。

### ただし FUNCTION のネストだけは禁止する

ブロック（IF/FOR/WHILE）は自由にネストできるが、**FUNCTION の中に FUNCTION を定義することは禁止**する
（FUNCTION はトップレベルのみ）。これはブロックのネストとは別問題で、以下の理由による。

- **MSXに表現手段がない** — MSX-BASIC は `GOSUB` の入れ子（呼び出しの入れ子）は可能だが、
  「関数の中の関数」という**入れ子スコープ**を表現する構文を持たない。
- **変換テーブルが二重スコープを扱えない** — 関数=GOSUBブロックの1対1対応を前提にしており、
  入れ子関数を許すとスコープが二重化してテーブル設計が破綻する。
- **逆変換が不安定になる** — 2文字名アロケータ（§5）の割当が入れ子スコープで二重化すると、
  逆引きが一意に定まらず復元が不安定になる。
- 以上より、**言語仕様として FUNCTION はトップレベル定義のみ**とし、入れ子関数は `E_NESTED_FUNCTION`
  として文法エラーにする（[03](03-lexer-parser.md)）。

## 5. 変換テーブルの役割（可逆変換の心臓部）

変換テーブル（`.map.json`）は単なる「可逆のためのメモ」ではなく、**MSX-BASIC を構造化BASICへ
正確に復元するための全対応情報**を保持する。変換時に生成し、逆変換時に参照する（[04](04-data-model.md#44-変換テーブル-maptable仕様3-7) にスキーマ定義）。

| 対応関係 | 順方向（変換時に記録） | 逆方向（復元に使用） |
|----------|------------------------|----------------------|
| **行番号 ↔ ラベル** | ラベル `@ADD` を行番号 `1000` に割当 | 行番号を削除しラベル/構造へ戻す |
| **変数名 ↔ 2文字MSX名** | `SCORE` → `S0`、`player_name$` → `A$`（2文字名アロケータ、`varNameMap`） | `S0` → `SCORE` に逆引き復元（スコープ込み） |
| **RETURN ↔ 戻り値** | `RETURN 式` → `FR=式 : RETURN` | 戻り値変数の受け取りを `RETURN 式` に戻す |
| **FUNCTION ↔ GOSUBブロック** | 関数を GOSUB 先ブロックへ展開 | GOSUB＋引数代入を `F(args)` 呼び出しに統合 |
| **REF ↔ 置換変数名** | `REF X` を呼び出し側の実変数名へ名前置換（ゼロコピー） | 置換された変数名を `REF X` に復元（複製ブロックは元関数へ統合） |
| **BREAK/CONTINUE ↔ ジャンプ先** | `loopId` 付きで NEXT直後/NEXT行へのGOTOを記録 | GOTO を内側ループの `BREAK`/`CONTINUE` に復元 |
| **IFの1行化/平坦化 ↔ ブロック** | `IF NOT()…GOTO` 等へ平坦化 | ブロック `IF…END IF`（ネスト含む）へ再構築 |
| **コメントの位置情報** | コメント原文と出力行位置を保持 | コメントを元の位置・原文のまま復元 |
| **エディタ行番号との対応** | 構造化ソース行 ↔ MSX行 を保持 | エラー表示・往復編集時の行対応に使用 |

> このテーブルが無い／壊れている場合、関数境界・REF・変数名の完全復元は不可能であり、
> 逆変換はベストエフォート（行番号削除＋大文字整形のみ）に縮退する（[06](06-reverse-transformer.md#610-逆変換の限界と注意)）。

### MSX変数名は「先頭2文字のみ有効」— 2文字名アロケータ

**MSX-BASIC の変数名は先頭2文字しか区別されない**（`COUNT`=`CO`、`_`は使用不可）。実質の名前数は
**型ごと約960個**（先頭英字×2文字目英数字、予約語除外）。構造化BASICの長い名前（`PLAYER_NAME$` 等）は
そのまま使えないため、本システムは**全変数を2文字MSX名へ割り当てる**。

- **型別プール**（`%` `!` `#` `$`）で割り当て。文字列も同じ規則（※以前の「26個(A$〜Z$)」は誤りで、文字列も約960個）。
- **再帰禁止＝非再入**なので、**生存区間が重ならないローカル変数は2文字名を使い回す**（レジスタ割当と同型）。
- グローバルはプログラム全体で固定の2文字名。割当結果は変換テーブルの **`varNameMap`** に記録し、逆変換で元名へ復元。
- ある型の約960名を使い切ると **`E_VAR_NAMES_EXHAUSTED`**。

#### `varNameMap`（変換テーブルの項目）

構造化の変数名と割当先2文字名（とスコープ・型）を保持する。

```
SCORE        → S0   (global, %)
player_name$ → A$   (global, $)
I (関数DRAW) → FI   (local,  %)
```

逆変換時はこのマップ＋スコープ情報を参照し、`S0` → `SCORE` のように復元する（同じ2文字名が別関数で
別変数に再利用されていても正しく逆引きできる）。スキーマは [04-data-model.md](04-data-model.md#44-変換テーブル-maptable仕様3-7)、
割当アルゴリズムは [05 §5.11](05-transformer.md#511-2文字msx名アロケータ全変数全型) を参照。

### 逆変換の限界（重要）

変換テーブルがあっても、逆変換は完全な往復一致（round-trip）を保証**しない**。以下は原理的な限界である。

- **変換テーブルが無い場合は完全復元不可** — 関数境界・REF・変数改名の情報が失われるため、
  行番号削除＋大文字整形のみのベストエフォートに縮退する。
- **最適化後の式は元に戻らない** — 定数畳み込み・複文化・整数化などの最適化を施した式は、
  最適化前の記述には戻せない（意味は等価だが字面は変わる）。
- **コメント位置が完全一致しない場合がある** — 複文化や行併合の影響で、コメントの行位置が
  元と1行ずれることがある（内容は保持）。
- **GOTO の手動編集は復元不能** — 変換後 `.bas` を手で編集して `.map.json` と乖離した行
  （特に手書きGOTO）は、構造へ畳み込めず `' [REVERSE?]` 注記付きでそのまま残す。

詳細は [06-reverse-transformer.md](06-reverse-transformer.md#610-逆変換の限界と注意) を参照。

## 6. EditorUI 要件（概要）

エディタは構造化BASICの編集と、変換後MSX-BASICの確認を担う。要件の詳細は [07-editor-ui.md](07-editor-ui.md)。

- **エディタ行番号の常時表示** — MSX行番号とは別の物理行番号を左ガターに常時表示する。
- **エラー行の × 表示** — 文法エラーのある行の右端に `×` を出す。
- **ツールチップでエラー内容** — `×` のマウスオーバーで該当行のエラー（行:列・種別・メッセージ）を表示する。
- **2つの表示タブ** — 「構造化BASICモード（編集可）」と「MSX-BASIC変換後モード（読み取り専用）」。
- **変換後モードは文法チェック通過時のみ表示** — エラー時は変換後コードを**更新しない**／表示しない。
- **保存ボタンの動作** —
  - 文法エラーあり → **変換前ファイルのみ保存**（変換不可）、`×`表示、「誤りがあります。確認してください」。
  - 文法エラーなし → 文法チェック → 変換 → 変換前保存 → 変換テーブル保存 → 変換後保存、「保存しました」。
- **WebMSX へのコピペ** — 変換後モードのコードはコピー可能で、**WebMSX 等のブラウザエミュレータへ貼り付けて即実行**できる形式（行番号付き・CRLF）で出力する。

### 変換後モードの更新タイミング

変換後（MSX-BASIC）の確定表示は **保存時** に更新する。それ以外はプレビュー扱いとする。

| 契機 | 変換後コードの扱い |
|------|--------------------|
| **保存ボタン押下時** | **正式更新**。文法チェック→変換→3ファイル保存まで行い、変換後タブへ反映する（基準動作） |
| 文法チェック実行時（入力停止後デバウンス） | エラー有無と `×` 表示のみ更新。変換後タブはプレビューとして再生成してよいが、ファイルは書かない |
| 入力のたびにリアルタイム変換 | **非推奨**（毎キーストロークでの全文変換は重く、未確定コードがちらつくため）。デバウンス（例400ms）でのプレビュー更新に留める |

> 確定（ファイル化される）変換後コードは常に「保存時点の文法エラーなしソース」と一致する。

## 7. 用語定義

| 用語 | 定義 |
|------|------|
| 構造化BASIC | 本システムの入力言語。FUNCTION・ブロック構造（自由ネスト可）・REF・BREAK/CONTINUE を持つ |
| MSX-BASIC | 変換後の出力言語。行番号・GOSUB/RETURN/GOTO ベース |
| 変換テーブル | 構造化BASICとMSX-BASICの全対応情報（§5）。逆変換に使用する |
| エディタ行番号 | UI上で常時表示する物理行番号（MSXの行番号とは無関係） |
| MSX行番号 | 変換後コードに自動割り当てされる行番号（100,110,…） |
| REF引数 | 参照渡し引数。MSX側では**呼び出し側の実変数名へ名前置換（ゼロコピー）**で実装する（スカラ・数値配列共通。文字列配列は不可） |
| ループスタック | ネスト許可下で BREAK/CONTINUE の対象（最も内側のループ）を解決する仕組み |

## 8. 実装前提と技術選定理由

| 選定 | 内容 | 採用理由 |
|------|------|----------|
| **実装言語：TypeScript** | フロント全体を TS で実装 | ① AST を判別共用体（discriminated union）で安全に表現・操作できる ② 型安全によりノード種別の取りこぼしをコンパイル時に検出 ③ TauriのWebView内でそのまま動作し、WebMSX連携にも適する（[10](10-tech-stack.md)） |
| **パーサー方式：再帰下降** | 各非終端記号に1パース関数 | ① 構造化BASICの文法（ブロックの再帰ネスト）と素直に対応し、ブロック本体を再帰呼び出しで表現できる ② エラー回復（panic-mode）を実装しやすく、複数エラーの同時報告に向く ③ ループスタック等の文脈をパース関数内で自然に持てる |
| **実行形態：Tauriデスクトップ（Win/Mac）** | OS標準WebView＋Rust。エディタ・変換・保存は両OS共通 | ① **軽量・低メモリ**（Chromium非同梱、Electron比でサイズ1/20程度）② TSコアをWebView内で無改変流用 ③ ネイティブのファイルI/O・外部プレイヤー起動が可能。再生3方式（A・B・C）は [10](10-tech-stack.md) |

その他の全体方針:

- エラーは **複数同時報告**（最初の1件で停止しない）。
- 大文字化はトークン段階で行うが、**文字列リテラルとコメントは対象外**。
- Domain層（Lexer/Parser/Transformer 等）は副作用を持たない純粋関数とし、テスト容易性を確保する（[02](02-architecture.md)）。
- **ターゲット機種**：既定 **turboR（MSX-BASIC 4.0）**。設定で最低対応機種を選べ、非対応/新しすぎる組み込みは警告（[12 §12.3](12-builtins.md#123-ターゲット機種フラグ)）。
- **組み込み命令・関数**は表で管理し**設定から編集・既定リセット可**（[12](12-builtins.md)）。**INCLUDE による複数ファイル分割**に対応（[01 §1.13](01-language-spec.md#113-include分割ファイル)）。
- **型**：`% ! # $`（既定単精度）、関数戻り値型は名前サフィックス。**演算子・優先順位はMSX-BASIC準拠**、`&H`16進、コメントは `'`/`REM`、配列はbase 0（[01 §1.11](01-language-spec.md#111-データ型演算子リテラル)）。

### テスト方針

- **ゴールデンテスト**：代表的な構造化ソース → 期待MSX-BASIC出力を固定し回帰検出。
- **往復テスト**：変換 → 逆変換 が構造的に一致することを検証（最適化で字面が変わる範囲は許容、[06 §6.10](06-reverse-transformer.md#610-逆変換の限界と注意)）。
- 各エラー/警告（`E_*`/`W_*`）の発火条件を単体テストで固定。

## 9. 変換後コードの品質要件

変換後の MSX-BASIC は機械生成だが、**人間が読める品質**を保証する（仕様8-3）。Formatter／Transformer が以下を満たす。

| 項目 | 規則 |
|------|------|
| **行番号** | `100,110,120,…`（既定10刻み）。関数ブロックは `1000,2000,…` のセグメント先頭から開始し、行帯で関数境界を可視化する |
| **インデント** | ネスト段数に応じて行番号の後を **2スペース単位**でインデントする。MSX-BASICはプログラムテキスト中の空白を保持するため可読性向上に有効（サイズ最優先時はオプションで抑制可、[09](09-optimization.md)） |
| **コメント** | 構造化側のコメントは**変換後も原文のまま保持**し、対応する位置へ出力する。関数境界には `' === FUNCTION X ===` を付与 |
| **大文字化** | キーワード・変数名・関数名は大文字化。**文字列リテラルとコメントは大文字化しない** |
| **可読性** | 行番号の飛び方を関数単位で規則化し、前方参照中心のGOTOで追いやすくする。1行が長くなりすぎないよう `:` 連結に上限を設ける |
| **実行可能性** | そのまま MSX-BASIC として実行可能（行番号昇順・重複なし・全ジャンプ解決済み）。WebMSX 等へ貼り付けて即実行できる |

### 記号の正規化ルール（コード例・出力で厳守）

本書のコード例および変換後出力は、以下の記号ルールを必ず守る。

| 記号 | ルール |
|------|--------|
| **ダブルクォート** | 必ず **ASCII の `"`** を使用する。スマートクォート（`“` `”`）は**絶対禁止**。シングルクォート（コメント `'`）も ASCII を使用 |
| **`:`（コロン）** | **文（命令）の区切り**。`A=1: B=2: RETURN` のように複数文を1行に連結する用途のみ |
| **`;`（セミコロン）** | **`PRINT` の連結区切り**。`PRINT "X=";N` のように出力を詰めて連結する。文の区切りには使えない |
| **`,`（カンマ）** | `PRINT` の**タブ区切り**（次のタブ位置へ）。文の区切り・連結には使えない |
| **改行コード** | 変換後 `.bas` は **CRLF**（WebMSX互換）。エディタ編集中の `.msxb` は LF |
| **文字コード** | MSXへ渡すファイル（`.bas` / `.msxb`）は **Shift-JIS（JIS X 0201＋0208、MSX実機に忠実）** で保存。`.map.json` のみ UTF-8（JSON標準）。Shift-JISで表現不能な文字は `E_NON_SJIS`。CP932は設定で任意選択可（[08 §8.6](08-file-save.md#86-ファイルフォーマット詳細)・[11 §11.13](11-editor-features.md#1113-エンコーディング改行)） |

> 要約：**文をつなぐのは `:`**、**PRINT で詰めて並べるのは `;`**、**PRINT でタブ揃えは `,`**。混同禁止。

### MSX-BASIC の 1 行 255 バイト制限（絶対制限）

**MSX-BASIC の 1 行あたりの最大長は 255 バイトである。** これは MSX-BASIC の内部構造（行バッファ長）が
255 バイトで固定されているためであり、変換後のコードがこの制限を超えると実行不能となる。
本システムは以下の仕様に従う。

1. 変換後の MSX-BASIC 行が 255 バイトを超える場合、**コンパイルエラー**とする。
   エラー例：**`E_LINE_TOO_LONG`**（MSX-BASIC の 1 行制限 255 バイトを超過）。
2. 255 バイトは「画面上の文字数」ではなく **「トークン化後のバイト数」** である。
   キーワードは 1 バイトに圧縮されるが、**文字列リテラルはそのままの長さでカウント**される。
3. 文字列リテラルを多用する `PRINT` 文、複文（`:` 連結）、長い式は特に制限に抵触しやすい。
   必要に応じて行分割を行うか、式を簡略化すること。
4. 変換器は変換後の行長を常に検査し、255 バイトを超える場合は**自動的に複文を分割**するか、
   （分割不能なら）**エラーとしてユーザに通知**する（自動分割・PRINT分割ロジックは
   [05-transformer.md](05-transformer.md#512-1行255バイト制限の検査と自動分割) / [09](09-optimization.md) を参照）。
5. 本制限は **MSX-BASIC の仕様上の絶対制限（ハード制限）であり、回避する手段は存在しない**。

> §9 の「複文の `:` 連結上限」「`PRINT` の区切り」は、この 255 バイト制限を満たすための運用指針でもある。
> 自動分割しても 1 文（分割できない単一の長い文字列リテラル・式）が 255 バイトを超える場合は `E_LINE_TOO_LONG`。

各品質要件の実装詳細は [05-transformer.md](05-transformer.md#57-行番号割当と最適化仕様3-6-8-3)・[09-optimization.md](09-optimization.md) を参照。

## 10. 変換ルールの要点

後続ドキュメントの理解を助けるため、特に効いてくる2つの変換ルールを概説する（詳細は [05](05-transformer.md)）。

### 10.1 関数セグメント（行番号レイアウト）

各 FUNCTION は **専用の行番号セグメント** に展開し、関数同士の行番号衝突を防ぐ。

- **セグメント幅** — 既定 **1000**。`MAIN` は `100〜`、以降の関数は `1000, 2000, 3000, …` から開始し、
  関数内部は10刻み（`1000,1010,1020,…`）。1関数あたり最大100文（1000÷10）を収容できる。
- **衝突回避** — 関数を別セグメントに隔離するため、関数内で行番号が重複・交差しない。ラベル→行番号は
  2パスで解決し、`GOSUB`/`GOTO`/`THEN <行>` を最後に数値化する。
- **関数数が増えた場合の拡張** — セグメント幅を超える大きな関数や多数の関数には、
  ① セグメント幅を可変（例 関数の文数に応じて 1000/2000）にする、② 刻みを10→5に詰める、
  ③ `lineStep`/`segmentStep`（[09](09-optimization.md) の `TransformOptions`）で調整する、で対応する。
- **行番号上限** — MSX-BASIC の最大行番号は **65529**。割当総量がこれを超えそうな場合は、
  刻み・セグメント幅を自動圧縮して上限内に収める。収まらない規模ならエラーとし、関数分割を促す。

### 10.2 BREAK / CONTINUE の変換（最も内側のループ）

ネスト許可のため、対象ループは **ループスタックで最も内側のループ** を特定して解決する。

- `BREAK` → 対象ループの **NEXT/WEND の直後** へ `GOTO`（ループ脱出）。
- `CONTINUE` → 対象ループの **NEXT/WEND 行（直前）** へ `GOTO`（次反復へ）。
- パーサーが各 `BREAK`/`CONTINUE` に `enclosingLoopId` を付与し、変換器が `loopId` 付きで
  `MapTable.controlFlow` に記録 → 逆変換で正しい内側ループの BREAK/CONTINUE に復元する。

詳細・多重ループ例は [05-transformer.md](05-transformer.md#56-break--continue--goto仕様3-4-3-5)。
