# 09. R800/R80最適化・追加要件

対応仕様: **【8. 追加要件】**

---

## 9.1 前提（仕様8-1）

変換により行数・GOSUB/GOTO が増えるため、**R800（turboR）/ R80** 系での実行を前提に最適化する。
MSX-BASIC はインタプリタであり、以下がボトルネックになりやすい:

- 行検索（GOTO/GOSUB の行番号探索）
- 変数名解決（変数テーブル線形探索）
- 行頭からの逐次トークン解釈

最適化はこれらの定数倍削減を狙う。

---

## 9.2 ネスト許可への方針転換と早期リターン（仕様8-2の再解釈）

> **【設計判断の根拠】** 原仕様8-2は「ネスト禁止によりコードがフラットになる」ことを前提にしていたが、
> 検討の結果、**ネストを許可した方が変換後コードの行数・メモリが小さくなる**と判断し、完全ネスト許可へ方針転換した。

### なぜネスト許可の方が出力が小さいか

ネストを禁止すると、入れ子が必要なロジックは FUNCTION へ切り出すしかなく、呼び出しごとに
**引数の値コピー・GOSUB・戻り値受け取り・REF書き戻し・専用変数**というボイラープレートが付く。
これが行数と変数テーブルを増やす。一方ネストを許可すれば、MSX-BASIC がネイティブに持つ
**ネスト可能な FOR/NEXT・1行 IF** に素直に乗せられ、出力が小さくなる。

| 同一ロジック | ネスト禁止（関数切り出し） | ネスト許可（直接変換） |
|--------------|---------------------------|------------------------|
| 配列の最大値 | 6行＋専用変数2個＋GOSUB | 3行・追加変数なし |

詳細な比較は本プロジェクトの設計判断ログ（README「設計変更の記録」）に対応する。

### 早期リターンは引き続き推奨（ただし必須ではない）

- ネスト許可後も **早期リターン**（`RETURN`）は可読性・性能の観点で推奨スタイルとする。
- ネストした IF 内からの `RETURN`/`BREAK`/`CONTINUE` を活用し、深い分岐の連鎖を避ける。
- 「正常系を直線的に、異常系を先に弾く」方針は整形の既定として維持する。

```basic
FUNCTION DIV(A, B)
    IF B = 0 THEN
        RETURN 0           ' 早期離脱（ネストしたIF内のRETURN）
    END IF
    RETURN A / B
END FUNCTION
```

### ネスト許可に伴うコスト（出力ではなく実装側）

ネスト許可で増えるのは**変換器・パーサーの複雑さ**であり、出力サイズではない。具体的には:

- ブロックスタック（ネスト構造の追跡）が必要。
- 多段 IF/ELSE の `GOTO` 平坦化ラベル管理。
- **BREAK/CONTINUE の飛び先**を「最も内側のループ」に解決するループスタックが必要
  （[05](05-transformer.md#56-break--continue--goto仕様3-4-3-5)）。
- ネスト段数が深いと MSX の FOR スタック（既定8段程度）・GOSUB スタックを消費するため、
  実行時スタック上限に注意（深い場合は一部を関数化して GOSUB に逃がす最適化を検討）。

## 9.3 行番号・ジャンプ最適化（仕様8-3）

### 9.3.1 行番号レイアウト

- 関数ごとに **1000刻みセグメント**（`1000`,`2000`,…）、内部は10刻み。
  - 行番号が単調増加 → MSXの行検索（多くの実装は先頭/現在行からの線形/二分探索）に有利。
- メイン（`@MAIN`）を先頭低位（100〜）に固定し、起動直後の実行開始を高速化。

### 9.3.2 GOTO/GOSUB 距離

- BREAK/CONTINUE/IF分岐の飛び先は **近傍**（同一関数セグメント内）に収める。
- 前方参照（後続行への GOTO）を基本とし、行検索の方向を一定化。

### 9.3.3 複文化（`:`）

- 連続する短い代入を `:` で1行にまとめ、**行数＝行ヘッダ解釈回数** を削減。
  - 例: `AA=1 : AB=2 : GOSUB 1000`（2文字名）
- 上限は2段階。**`maxLineLength`（既定80）= 可読性のためのソフト上限**、
  **255バイト = MSX-BASICのハード上限（絶対制限）**。後者を超える行は自動分割し、
  分割不能なら `E_LINE_TOO_LONG`（[05](05-transformer.md#512-1行255バイト制限の検査と自動分割)・[README §9](README.md#msx-basic-の-1-行-255-バイト制限絶対制限)）。

### 9.3.4 インデント／可読性（仕様8-3）

- MSXのリスト上ではインデント不可だが、**関数境界コメント** `' === FUNCTION X ===` で可読性確保。
- 行番号の飛び方を関数単位で規則化し、リストを追いやすくする。

## 9.4 変数最適化

### 9.4.1 型サフィックス

| 用途 | サフィックス | 効果 |
|------|------------|------|
| ループカウンタ・整数 | `%`（整数型） | 浮動小数より高速（R800で顕著） |
| 文字列 | `$` | 必須 |
| 既定 | （単精度） | — |

- ループ変数や添字は整数型 `I%` を優先採用（変換オプション）。

### 9.4.2 変数名（2文字アロケータ）と DEFINT/DEFSTR

- MSX変数名は**先頭2文字のみ有効・`_`不可**。全変数を **2文字MSX名アロケータ**（型別プール・生存解析、
  約960個/型、`varNameMap`）で割り当てる（[05 §5.11](05-transformer.md#511-2文字msx名アロケータ全変数全型)・[01 §1.10](01-language-spec.md#110-変数スコープ)）。
  旧 `AD_`/`SW_` プレフィクス方式・26文字列スロットは廃止・統合済み。
- よく使う変数を変数テーブル前方に置くと解決が速い（`DEFINT I-N` などの活用も検討）。
- **`DEFINT`/`DEFSTR` は構造化BASIC側では記述できない**（言語仕様に含まれない）。
  したがってこれらは **変換後コードへ変換器が挿入する最適化オプション**として扱う
  （既定オフ。有効時は各関数セグメント先頭等へ `DEFINT I-N` 等を自動挿入する）。

### 9.4.3 戻り値変数

- 戻り値は関数ごとに2文字名1つ（例 `FR`）。再利用可能で変数増加を抑制。

## 9.5 関数展開戦略（インライン vs GOSUB）

| 戦略 | 利点 | 欠点 | 適用 |
|------|------|------|------|
| GOSUB展開（既定） | コードサイズ小 | GOSUB/RETURNのオーバヘッド | 既定・多用関数 |
| インライン展開（オプション） | ジャンプ無しで高速 | サイズ増 | 小さく超高頻度の関数 |

- 既定は GOSUB（仕様3-1）。ホットパスの極小関数のみインライン化オプションを提供。
- インライン時もローカル変数の名前は2文字アロケータ（[04 §4.7](04-data-model.md#47-命名規約変換で使用)・[05 §5.11](05-transformer.md#511-2文字msx名アロケータ全変数全型)）が衝突なく割り当てる。

## 9.6 REF（名前置換）と配列値渡しのコスト（[05 §5.4](05-transformer.md#54-ref引数--名前置換方式仕様3-3) 補足）

REF は **名前置換方式（ゼロコピー）**。コピーバックは廃止。よってREF自体のコピーコストは無い。

- **スカラ・配列のREF**：呼び出し側の実変数名へ置換するだけ＝**コピーゼロ**。最速。
- **コスト＝コード複製**：同じ関数を**異なる配列名**で呼ぶと、配列名ごとに本体を複製する（§5.4.3）。
  - ハード上限は無いが、**異なる配列10個以上**で `W_REF_MANY_VARIANTS`（warning）を出す。
  - 複製を抑えたいなら、同じ関数に渡す配列を絞る／処理を共有グローバル配列に寄せる、等で調整。
- **配列の値渡し（REF無し）**：全要素コピー（O(n)）で重い。
  - **32要素以上**の値渡しは `W_ARRAY_VALUE_COPY`（warning）。禁止はしない（プログラマの選択）。
  - 速度が要るなら **REF（ゼロコピー）を使う** のが定石。値渡しは「呼び出し側を壊したくない」意図のときのみ。
- **文字列配列のREFも可**（名前置換で数値配列と同一。旧禁止は26スロット前提の誤りで撤廃）。

> いずれの警告も `transformErrors` に **warning** として積み、保存・変換はブロックしない（[08 §8.4](08-file-save.md#84-savecontroller-実装02-と整合)）。

## 9.7 最適化オプション（変換設定）

変換時に切替可能な設定を持つ。

```ts
interface TransformOptions {
  lineStep: number;          // 既定 10
  segmentStep: number;       // 既定 1000（関数セグメント間隔）
  joinWithColon: boolean;    // 複文化（既定 true）
  maxLineLength: number;     // 複文化のソフト上限（既定 80）。ハード上限255は別途強制（E_LINE_TOO_LONG）
  indentSpaces: number;      // ネスト1段あたりのインデント空白数（既定 2、0で無効化＝サイズ優先）
  whileNative: boolean;      // WHILE/WEND ネイティブ出力（既定 true）
  integerLoopVars: boolean;  // ループ変数 %（既定 true）
  inlineHotFns: string[];    // インライン化する関数名
  refManyVariantsWarn: number;   // REF複製の警告閾値（既定 10：異なる配列数）
  arrayValueCopyWarn: number;    // 配列値渡しの警告閾値（既定 32：要素数）
}
```

## 9.8 最適化と可逆性の両立

- いかなる最適化を行っても、**MapTable に十分な復元情報を残す**ことを必須とする（仕様3-7/4）。
- 複文化・インライン化・整数化は、逆変換で元の構造化表現へ戻せる範囲でのみ適用する。
- 戻せない最適化（例: 配列共有でREF痕跡が消える）は、MapTable に専用フラグを置いて補う。

## 9.9 計測指針（任意）

- 変換後コードについて、行数・GOSUB数・GOTO平均距離・変数数をレポート出力する開発支援機能を推奨。
- WebMSX 上でのベンチ（フレーム数/実行時間）を比較し、最適化オプションの効果を検証する。
