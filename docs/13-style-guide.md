# 13. FunctionBASIC スタイルガイド

構造化BASICには、PHP の PSR や Python の PEP 8、Go の gofmt に相当する**公式のコーディングスタイル標準が歴史的に存在しなかった**（言語規格 ANSI/ISO BASIC は構文・意味論の標準であってスタイル規約ではない）。本章は、その空白を埋める **FunctionBASIC の正規コーディングスタイル**を定義する。

**手本**（既存の実績ある標準の合成。ゼロからの発明ではない）:

- **骨格＝Microsoft「Visual Basic Coding Conventions」** — VB は構文がほぼ同型（`If…End If` / `For…Next` / `Do…Loop` / `Select Case…End Select` / `Sub`・`Function`）。レイアウト・1文1行・コメント規約を流用する。
- **ブロックインデントの厳密さ＝「Ada Quality and Style Guide」（＋ `gnatpp`）** — `if…end if` / `loop…end loop` / `case…end case` という「閉じキーワードで終わるブロック」が構造化BASIC と 1:1。
- **思想＝gofmt / PSR-12 / Black** — 「唯一の正規形・オプション無し・整形器がリファレンス実装」。bikeshed（空白論争）を消す。
- **casing だけ FB 独自（全大文字）** — 理由と、VB の case 依存規約をどう置換したかは [§13.5](#135-手本にした標準と設計判断)。

**用語**：**必須(MUST)** = 従わなければならない／**推奨(SHOULD)** = 特段の理由がなければ従う／**任意(MAY)** = 好みで選べる。

**二層構成**：

- [§13.2](#132-整形器が保証する規則自動整形) … **整形器が自動で保証する規則**。整形（`Ctrl/Cmd + Alt + L`）はこの層の**リファレンス実装**。
- [§13.3](#133-記述者が守る規約手動) … **記述者が守る規約**（自動整形されない）。

## 13.2 整形器が保証する規則（自動整形）

整形コマンドを実行すると、以下が自動的に適用される。**整形器の出力＝本節の定義**であり、両者が食い違えばそれはバグ（[§13.4](#134-整形器リファレンス実装)）。

### R1. 大文字化（casing）［必須］
- コードのトークン — **キーワード・組み込み命令/関数・変数・関数・`STRUCT` 型名とフィールド・`MACRO`/`DATASET` 名・ラベル・数値リテラル（`&H` 等）** — は**すべて大文字**にする。
- **文字列リテラル（`"…"`）とコメント（`'…` / `REM`）は変換しない**（原文の大小をそのまま保持）。
- FB は識別子が**大小文字を区別しない**ため、大文字が唯一の正規形。**PascalCase / camelCase は使わない**。→ [01 §1.6](01-language-spec.md#16-自動大文字化仕様1-6)

### R2. インデント［必須］
- 1 段 = **半角スペース 4**。タブはスペースへ展開。
- **ブロックを開く**キーワードで本体を +1 段：`FUNCTION` / ブロック `IF…THEN` / `FOR` / `WHILE` / `DO` / `SELECT CASE` / `STRUCT` / `DATASET` / `EVENT`。
- **ブロックを閉じる／中間**キーワードは -1 段でその行を字下げ：`END *` / `NEXT` / `WEND` / `LOOP`（閉じ）、`ELSE` / `ELSEIF` / `CASE`（中間）。
- `SELECT CASE` は `CASE` 行を +1 段、その本体をさらに +1 段。
- `ASM … END ASM` の**本体は原文のインデントのまま**（アセンブリのラベル桁を壊さない）。`ASM` / `END ASM` 行だけブロック段に合わせる。

### R3. 1 文 1 行［必須］
- 1 行に 1 文。`:` による複文は不可（言語規則。[01](01-language-spec.md)）。1 宣言 1 行。

### R4. 空白［必須］
- 行内のトークン間の**連続空白は 1 個に詰める**（`FOR     I` → `FOR I`）。元が無空白なら無空白のまま（`X=X` は `X=X`）。
- **行末の空白は除去**。空行にはインデントを付けない。
- **文字列リテラルの中身は変更しない**（`"a    b"` はそのまま）。
- **末尾コメント直前の桁揃え空白は温存**する（R5）。

### R5. コメント［必須］
- **行頭コメント**（行が `'` / `REM` で始まる）：先頭の空白だけブロック段へ正規化し、**`'`/`REM` 以降の本文（内部の空白も）は変更しない**。
- **末尾コメント**（コード + `'…`）：**コード部だけ**整形し、**コメント部は桁揃えの空白ごと変更しない**。

> **トレードオフ（gofmt と同じ思想）**：R1・R4 は**意味を変えない**（整形前後で変換後 MSX-BASIC は不変）。R4 は連続空白を詰めるため、**意図的な桁揃え（`=` の縦揃え等）も詰まる**。唯一の正規形を優先する設計判断であり、桁揃え保持モードは持たない。

## 13.3 記述者が守る規約（手動）

自動整形されないが、FB の「家の流儀」。

### N1. 命名［推奨］
- 全大文字。語の区切りは**アンダースコア**（SCREAMING_SNAKE_CASE）：`PLAYER_X%` / `NAME_TABLE_BASE%` / `GEN_NEXT`。
  - ※ 大文字化で PascalCase の**語境界表示が失われる**ため、可読性はアンダースコアで確保する（VB の「`_` を避ける」規則は FB では**逆**になる）。
- **型サフィックス** `% ! # $` を付ける（`STRICT` では必須）。
- 意味の分かる名前を付ける。1〜2 文字の使い捨ては短命なループ変数など局所に限る。

### N2. 役割の示し方（casing を使わない代わり）［推奨］
VB は casing で識別子の役割（型/メンバ/ローカル/定数）を表すが、FB は全大文字なので**別の手段**で示す：

| 役割 | 示し方 |
|---|---|
| 型 | 型サフィックス `% ! # $` |
| 定数 | `CONST` 宣言 |
| グローバル | `GLOBAL` 宣言 |
| 構造体 / マクロ / 名前付きデータ | `STRUCT` / `MACRO` / `DATASET` |

### N3. ブロック構造［必須／推奨］
- `IF` は**ブロック形式のみ**（`IF 条件 THEN 文` の 1 行 IF は不可 → `E_IF_SINGLE_LINE`）。多分岐は `ELSEIF` または `SELECT CASE`。
- ループは `FOR` / `WHILE` / `DO…LOOP`（前判定・後判定・無限）。脱出は `BREAK` / `CONTINUE`。
- `FUNCTION` のネストは禁止。その他のブロックは自由にネストしてよい（[README §2](README.md)）。

### N4. 宣言の位置［推奨］
- `GLOBAL` は、それを使う**各 `FUNCTION` の先頭**で宣言する（トップレベルでも宣言）。局所変数は宣言不要（代入で生成）。

### N5. 演算子・区切り（言語差＝VB を真似ない箇所）［必須］
- **文字列連結は `+`**（VB の `&` ではない）。MSX-BASIC 準拠。
- **`PRINT` の項目区切りは `;`**。これは連結の `+` とは**別物**（`;` は表示の区切り、`+` は文字列値の連結）で、相互に書き換えない。
- 文字列の引用符は **ASCII の `"`** のみ（スマートクォート禁止）。

### N6. コメント［推奨］
- `'` を推奨（`REM` も可）。ファイル・関数の先頭にヘッダコメントを置く。
- 桁揃えした末尾コメントは整形器が保持する（R5）。

### N7. 厳格化ディレクティブ［推奨］
- 非自明なプログラムでは `STRICT`（静的型チェック、[01 §1.15](01-language-spec.md)）と `OPTION EXPLICIT`（未宣言変数の検出、[01 §1.16](01-language-spec.md)）を付ける。

### N8. ファイル構成［推奨］
上から順に：**ヘッダコメント → `STRICT`/`OPTION EXPLICIT` → `CONST` → `MACRO` → `STRUCT` → `GLOBAL` 宣言群 → `FUNCTION` 群 → MAIN 本体 → 末尾に `DATASET`/`DATA`**。（`examples/` はこの構成）

## 13.4 整形器＝リファレンス実装

- 整形（`Ctrl/Cmd + Alt + L`）を実行した結果が [§13.2](#132-整形器が保証する規則自動整形) 準拠。**整形器の出力と本章が食い違えば、それはバグ**（どちらかを直す）。実装は Formatter（[02](02-architecture.md)、`editor/app.js` の `formatSource`）。
- gofmt と同様、**唯一の正規形・オプション無し**。桁揃えは保持しない。
- **意味は変えない**（casing と空白のみ）。整形の前後で**変換後 MSX-BASIC は不変**（バンドル `examples/` 全件で検証）。

## 13.5 手本にした標準と設計判断

- **なぜ独自標準が要るか**：構造化BASIC は方言が分裂し（MSX-BASIC / QuickBASIC / FreeBASIC / VB …）、PSR/PEP 8 を出す単一のコミュニティ権威が無かった。言語規格（ANSI X3.60-1978 / X3.113-1987・ISO/IEC 10279:1991）は構文の標準でスタイル規約ではない。唯一の事実上の共通慣習は「**予約語は大文字**」で、これはトークン化エディタ（MSX-BASIC / QuickBASIC 等）が入力時に自動大文字化した副産物。FB はそれを**全識別子へ拡張**した。
- **casing を全大文字にしても VB 規約と矛盾しない**理由（調査結果）：VB も case **非依存**言語なので、正規 casing は純粋なスタイル選択で意味を持たない。casing に依存する VB 規約は 2 つだけで、いずれも FB の機構へ無矛盾に置換できる：
  1. **PascalCase による語境界表示（＋ `_` 回避）** → 全大文字では語境界が消えるため、**アンダースコア区切り**で置換（N1）。VB の「`_` を避ける」は FB では反転する。
  2. **casing による役割表現**（型=Pascal / ローカル=camel 等） → **型サフィックス＋宣言キーワード**で置換（N2）。

  これらは FB の既存コード（`examples/`）が既に実践しており、後追いの明文化にすぎない。頭字語の casing 規則（`IOStream` 等）は全大文字で無意味化し、そのまま消える。
- **VB を真似ない箇所（言語差、casing とは無関係）**：連結は `+`（`&` でない）、型はサフィックス（`As Type` でない）、`PRINT` 区切りは `;`（N5）。
