# 相談メモ：構造化BASIC→MSX-BASIC変換における「配列の参照渡し(REF)」の実装

> **【決着済み・2026-06】** 本メモは相談用スナップショット（審議の記録）。
> 結論として **REF＝名前置換方式（ゼロコピー）に統一**、コピーバックは廃止、文字列配列REFは禁止、
> 配列値渡しは許可（巨大時に警告）で確定した。現行の正式仕様は
> [01 §1.3.3](01-language-spec.md#133-引数渡し仕様1-3)／[05 §5.4](05-transformer.md#54-ref引数--名前置換方式仕様3-3)／[06 §6.5](06-reverse-transformer.md#65-ref-復元仕様4-1)／[09 §9.6](09-optimization.md) を参照。
> 以下は決定に至るまでの検討内容（歴史的記録）。

本メモは第三者に相談するための自己完結サマリ。前提知識なしで読める粒度でまとめる。

---

## 1. 背景（何を作っているか）

- **構造化BASIC**（`FUNCTION`・ブロック構造・`REF`引数・`BREAK`/`CONTINUE` を持つ自作言語）で書いたコードを、
  **MSX-BASIC**（行番号・`GOSUB`/`GOTO`ベース）へ機械変換する**エディタ兼トランスパイラ**を設計中。
- `FUNCTION` は MSX-BASIC の **`GOSUB` ブロック**に展開する。
- 引数渡しは **無印＝値渡し／`REF`＝参照渡し**。

## 2. 前提・制約（MSX-BASICの性質）

- **変数はすべてグローバル**。ローカルスコープは存在しない。
- **ポインタ・参照（配列の別名）が無い**。実行時に「この変数名を指す」という間接参照ができない。
- 1行 **255バイト**制限、文字列変数は **A$〜Z$ の26個**まで、等の厳しい制約あり（本論点とは別だが関連）。
- 「ローカル変数」は **改名で疑似実現**している。例：関数 `ADD` 内の `A` → グローバル名 `AD_A` に改名して衝突回避。

## 3. すでに確定している設計

- **スカラの値渡し**：呼び出し前に `AD_A = <実引数>` を代入（コピー1命令、安価）。
- **スカラのREF（参照渡し）**：**コピーバック方式**。呼ぶ前に `tmp = X`、`GOSUB`、戻った後に `X = tmp`。O(1)で安価。
- **再帰は禁止**（固定変数名を使い回すため、再入すると壊れる。呼び出し循環を検出してエラー）。
- `REF` の実引数は **変数名のみ**（式・即値は禁止）。→ **どの変数を指すかはコンパイル時に必ず確定する。**

## 4. 論点：配列のREFをどう実装するか

スカラと違い、**配列はコピーが O(n)（全要素ループ）**になる。素直に「スカラと同じコピーバック方式」を配列に当てると：

| 配列の渡し方 | MSX側のコスト |
|---|---|
| 値渡し | コピーイン 1ループ（O(n)） |
| REF（コピーバック） | コピーイン＋コピーアウト＝**2ループ（O(n)×2）** |

→ **問題点：コピーバック方式だと「配列のREF」は値渡しより遅い。**
「速くしたいからREFを付ける」という一般的な直感（REF＝コピーしない＝速い）と逆になり、**気持ち悪い／非一貫**。

### 設計上の要望（依頼者の方針）

- **言語仕様は一貫させたい**：「無印＝値渡し／REF＝参照渡し」をスカラも配列も同じ規則で通す。
- **コストを払うかはプログラマが判断**（REFを付けるかどうかで決める）。配列だけ暗黙にグローバル化、のような特例は避けたい。
- したがって **REFは「本当の参照＝ゼロコピー＝速い」であってほしい**。

## 5. 現状の有力案：REF＝「呼び出し側のグローバル名へ置換」

MSXは全変数グローバルなので、**REFした仮引数を、呼び出し側の実際の（プリプロセス後の）グローバル配列名にそのまま置換**すれば、
コピー無しの真の参照になる。REF実引数は変数名限定なので、置換先は常に確定する（§3）。

### ただし制約：「1つのGOSUB本体には1つの名前しか焼き込めない」

ポインタが無いため、単一の関数ブロックは固定の配列名しか参照できない。したがって：

- **常に同じ配列で呼ぶ関数** → 仮引数をその配列名に置換し、**GOSUBブロック1個**（複製なし・ゼロコピー）。理想。
- **異なる配列で呼ぶ関数** → 1本のブロックに複数名を入れられないので、**配列名ごとに専用ブロックを複製生成**
  （= モノモーフィック化／インライン相当）。**コピーループは無い**。増えるのは本体の行だけ。

### 具体例（異なる配列で呼ぶケース）

構造化BASIC:
```
FUNCTION SUM(REF A, N)
    FOR I=1 TO N: LET S=S+A(I): NEXT
    RETURN S
END FUNCTION

LET T1 = SUM(REF SCORE, 10)
LET T2 = SUM(REF DAMAGE, 5)
```

変換後 MSX-BASIC（配列名ごとにブロック複製。局所変数 SUM_S/SUM_I/SUM_R は非再帰なので共有可）:
```
130 GOSUB 2000: T1=SUM_R        ' SCORE用
140 GOSUB 3000: T2=SUM_R        ' DAMAGE用
...
2000 SUM_S=0
2010 FOR SUM_I=1 TO 10: SUM_S=SUM_S+SCORE(SUM_I): NEXT
2020 SUM_R=SUM_S: RETURN
3000 SUM_S=0
3010 FOR SUM_I=1 TO 5: SUM_S=SUM_S+DAMAGE(SUM_I): NEXT
3020 SUM_R=SUM_S: RETURN
```

### この案の性質

- **言語規則は完全一貫**（値=コピー／REF=参照、スカラも配列も同じ）。
- **REFが本当に速度レバー**になる（ゼロコピー）。
- コストは **コード複製のみ**（その関数を“異なる配列”何種類で呼ぶか＝複製数）。同じ配列だけなら複製ゼロ。
  → 「コストはプログラマが判断」という方針と一致。
- 複製で増えるのは本体行のみ。局所変数・戻り値変数は共有（非再帰前提なので安全）。

## 6. 各案の比較（まとめ）

| 案 | 一貫性 | 配列REFのコスト | 汎用性（同関数を別配列に） | 代償 |
|----|:---:|---|:---:|---|
| (1) コピーバック（スカラと同方式） | ○ | **遅い**（コピーイン+アウト） | ○ | 速度・サイズ・255制約に不利。REF＝速いが成立しない |
| (2) 名前置換＋必要時のみブロック複製 ★有力 | ◎ | **ゼロコピー** | ○ | 異なる配列の数だけ本体を複製（コード増） |
| (3) 配列は暗黙にグローバル共有（引数にしない） | ✕ | ゼロコピー | △ | 言語が非一貫（配列だけ特例）。依頼者却下 |

★ 現状は **(2)** を有力案としている。

## 7. 残っている決定事項（相談したいこと）

1. **スカラのREFも (2) の名前置換に統一するか**、現状の **コピーバックのまま**にするか。
   - 意味（参照渡しの挙動）は同じ。違うのは実装とコード量だけ。
   - スカラはコピーバックが安価＆複製不要なので、「配列だけ置換」にすると実装は型で分岐するが、
     **言語仕様としては一貫**（プログラマから見た意味は同一）。実装の非対称を許容してよいか。
2. **(2) のブロック複製を、無制限に許すか上限を設けるか**。
   - 同じ関数を多数の異なる配列で呼ぶとコード爆発の可能性。警告/上限/インライン抑制の要否。
3. **多次元配列・文字列配列**でも (2) が成立するか（成立する見込みだが要確認）。
   - 文字列配列は A$〜Z$ の26スロット制約とも絡む。
4. **値渡しの配列**（REF無しで配列を渡す＝関数が自分用コピーを持つ）を許可するか、
   それとも「配列は実質REF前提」にするか。値渡し配列はコピーイン O(n) のコストがかかる。

## 8. 補足（関連する確定事項）

- 関数は基本 `GOSUB` 展開、再帰禁止、行番号は関数ごとに1000刻みセグメント。
- REF実引数は変数名限定（式・即値禁止）＝置換先が常に確定する根拠。
- これらは確定済みで、本論点（配列REFの実装方式）のみ未決。
